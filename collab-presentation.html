Designer prompts

🎨 Designer — What to tell Claude Code

Your job is layout, color, and feel. Claude Code outputs HTML. You iterate until it looks real.
🖼️
Start with structure
"Design a clean one-page UI for [our tool]. It needs: a header, an input area where users type [X], a button that says [Y], and a results section below. Use a dark background, teal accent color, and modern card-style layout. Output as one self-contained HTML file."
Designer
🎨
Iterate on feel
"The layout is good but it feels generic. Make it feel more premium - refine the typography, add subtle gradient accents, improve spacing. Keep the same structure."
Designer
📦
Hand off to Builder
Save the HTML file. Tell the Builder: "The design is done. The button doesn't do anything yet - that's your job. The results section is empty - fill it with real output."
Builder next

Builder + Detailer prompts

⚙️ Builder then ✍️ Detailer

Builder makes it work. Detailer makes it good. Sequential but fast - both running Claude Code simultaneously on their part.
⚙️
Builder: wire up the logic
"Here's the HTML file the designer built [paste it]. The button does nothing. Make it so when clicked, it takes the user's input and sends it to Claude API with this system prompt: [paste your prompt]. Show the response in the results section. Keep all the existing design."
Builder
🔁
Builder: fix what breaks
"The output shows up but it's raw text, no formatting. Parse it as markdown and render it properly. Also add a loading spinner while the API call is running."
Builder
✍️
Detailer: polish the output
"The tool works but the output sounds robotic. Rewrite the system prompt so the response sounds like a smart human wrote it, not an AI. Then write 5 realistic test inputs we can use for the live demo and show me the output for each."
Detailer
When Claude Code slows down

Hit a rate limit?
Switch lanes, keep moving.

Claude Code has usage limits. When it slows down or stops, don't wait - jump to Claude web and keep the work going in parallel.
⚠️
You hit a rate limit in Claude Code
Terminal slows down or says you've hit a usage limit. This is normal. Don't stop.
Signal
🌐
Open Claude web — claude.ai
Different usage pool. Free to use. You can keep building here while Claude Code recovers.
"I'm building [describe your tool]. Here's where we're up to: [paste your notes or current code]. Help me plan the next steps and mock up the frontend."
Claude web
🎨
Ask Claude web to mock up the frontend
Claude web can generate full HTML/CSS. Use it to get the design done while the terminal is paused.
"Build a clean frontend for [our tool]. It needs: [describe inputs, outputs, layout]. Output a single self-contained HTML file I can paste into my project."
Claude web
📋
Copy the code — paste it into Claude Code
When Claude Code is ready again, grab everything Claude web generated and hand it over.
"Here's the HTML file Claude web built for our frontend [paste it]. Now wire up the logic so the button actually calls [our process] and shows results in the output section."
Claude Code
Two tools, one team

Claude web vs
Claude Code — use both

They're not the same tool. They have different strengths. Smart pods use both at the same time.
🌐 Claude web (claude.ai)
Planning - "what should we build next?"
Mocking up frontend designs
Writing and refining your prompts
Drafting notes.md and presentation copy
Brainstorming edge cases to test
Works when Claude Code is rate limited
⚙️ Claude Code (terminal)
Building - writes and runs actual files
Wiring up logic, APIs, and interactions
Pushing work to the pod GitHub repo
Reading your CLAUDE.md for context
Iterating on code files directly
Running things, seeing real output
💡
The relay strategy
Designer uses Claude web to mock the UI. Builder pastes that into Claude Code to wire logic. Detailer uses Claude web to refine copy and test prompts. Nobody waits. The work moves forward on both tracks simultaneously.
Seeing your app

Want to see it?
Just ask.

Claude Code handles the rest. You don't need to know what a "server" is or how to run one. One sentence is enough.
Say this to Claude Code
"Open this in my browser so I can see it."
That's it. Claude Code will start a local server and give you a link like http://localhost:3000 — click it and your app appears.
📄
Plain HTML file
Claude Code opens it directly. No server needed. Instant.
⚡
App with logic
Claude Code starts a local server automatically and gives you the link.
🔄
Made a change?
"Refresh my browser" or just hit refresh - Claude Code keeps it live.
If it doesn't open automatically: look in the terminal for a line that says localhost:XXXX and paste that URL into your browser.
Halfway there

Midpoint check-in —
one person, two minutes.

Not a full demo. Just a quick pulse check. Pick one person from your pod to speak. Everyone else listens and takes notes.
Cover this
🏗️
What you built so far
One sentence. What does it do right now, even if it's rough.
✅
What was easy
What did Claude just nail on the first try? What surprised you?
🧱
What blocked you
Where did you get stuck? Rate limits, wrong output, unclear prompts?
🎯
What's next
One specific thing you're building in the next 30 minutes.
Prompt to prep
"Help me prepare a 2-minute
midpoint update for our pod.

Here's where we're at:
[paste your notes.md]

Write a quick script covering:
- What we built so far
- What was easy
- What blocked us
- What we're doing next

Keep it honest and specific.
No corporate speak."
⏱️
Two minutes max. If you're going longer, you're explaining too much. The point is the blocker — say it fast so the room can help.
After each pod speaks
→ Anyone with the same blocker raises their hand
→ Anyone who solved it shares how
→ Move on - don't get stuck in one pod's problem
Turn your work into a presentation

NotebookLM —
pitch deck + explainer video

You built something. Now make it presentable. NotebookLM reads your repo and generates a polished audio pitch and slide outline in minutes.
How to do it
1
Go to notebooklm.google.com
Sign in with Google. Click New Notebook.
2
Upload your sources
Click Add Source - paste your GitHub repo URL, or paste the contents of your notes.md and README.md as copied text.
3
Generate Audio Overview
Click Audio Overview in the right panel. Click Generate. Two AI hosts discuss your project like a startup pitch. Takes ~2 minutes.
4
Generate the pitch deck outline
In the chat box, paste the prompt on the right. Copy the output into Google Slides or Canva.
Pitch deck prompt
"Create a pitch deck outline
for our solution. Structure:

Slide 1 - Title + what we built
Slide 2 - The problem + who has it
Slide 3 - Our solution in plain English
Slide 4 - How it works (3 steps)
Slide 5 - Live demo [we show live]
Slide 6 - What we'd build next
Slide 7 - What surprised us today

2-3 bullet points per slide.
Punchy, specific, no corporate speak."
FOR THE EXPLAINER VIDEO
1. Play the Audio Overview
2. Hit Shift+Cmd+5 (Mac) or Win+G (Windows) to record your screen
3. While audio plays, show your app or GitHub repo
4. Stop recording — you have a video

Or use loom.com for instant shareable link
💡
Make sure notes.md is filled in before you open NotebookLM — it can only work with what you give it.

The handoff

Passing work without
losing context

The #1 mistake: handing off a file with no explanation. Claude Code on the other end has no idea what's done, what's broken, or what not to touch.
✗ Bad handoff
Paste a file and say "here you go"
No context on what's done vs what's missing
Builder rewrites the designer's work
Detailer doesn't know which output is the right one to test
Everyone redoes each other's work
✓ Good handoff
Paste the file + 3 sentences: what's done, what's missing, what not to touch
"The design is locked. Only add to the JS. Don't change the CSS."
"The button works. The output formatting is broken - that's the only thing to fix."
"Test with these 3 inputs: [list them]. The 3rd one fails."
Next person knows exactly where to start
